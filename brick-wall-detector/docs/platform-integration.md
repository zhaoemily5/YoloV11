# 2.4 智能化查勘平台开发与数据资产集成

为了将 YOLOv11 红砖墙病害识别模型真正转化为施工现场可用、管理部门可查、后续研究可迭代的数据化工具，课题组开发了“红砖墙病害智能检测 Web 系统”。平台围绕历史建筑清水砖墙查勘业务，构建了从“规范拍摄—图片上传—AI 识别—结果可视化—面积换算—修缮报告—数据归档—系统审计”的完整业务闭环。系统部署后可通过浏览器直接访问，现场人员无需配置专业算法环境，即可完成红砖墙病害的快速识别、定量统计和报告输出。

该平台并非单一算法演示页面，而是面向真实工程应用开发的全栈式查勘系统。前端采用 Vue 3、TypeScript、Element Plus、Canvas、ECharts 等技术构建交互界面；后端采用 Node.js、Express、Multer、Sharp、Python 推理脚本及 ONNX Runtime 预留模块构建服务接口；模型层集成 YOLOv11 红砖墙病害检测权重，并统一封装为 RESTful API；数据层围绕用户、历史检测记录、系统日志、系统配置、立面任务和报告结果建立轻量化持久化机制。平台支持裂缝、缺损、植物附着、风化、泛碱五类病害的统一识别、展示、统计和报告生成，为课题成果的工程化落地提供了重要支撑。

## 2.4.1 平台定位与业务目标

本平台的建设目标是解决传统红砖墙病害查勘中“采集不规范、识别靠经验、面积难量化、记录难归档、报告整理慢”的问题。传统现场查勘通常依赖人工目测和纸质记录，病害类型判断、严重程度描述和修缮范围估计均存在较强主观性；即使引入算法模型，若缺少标准化 Web 平台，也往往停留在研究人员本地运行脚本、手动整理结果的阶段，难以支撑工程项目日常使用。

针对上述问题，平台在功能定位上形成了四类核心能力。第一，提供面向一线人员的单图敏捷查勘能力，使工作人员能够上传手机或相机拍摄的墙面照片，快速获得病害识别结果。第二，提供面向管理人员的整墙立面普查能力，对正射立面图或全景大图进行切片分析、网格统计和热力图展示。第三，提供面向修缮决策的报告生成能力，将病害检测结果自动转化为包含总体风险、病害详情、修缮建议和优先级计划的结构化报告。第四，提供面向数据资产管理的历史记录、系统日志和配置管理能力，使每次查勘结果均可追溯、可复核、可导出。

从课题研究角度看，平台承担了“算法模型工程化封装”和“病害数据资产集成”两项任务。一方面，它将 YOLOv11 模型从实验环境中的训练权重转化为浏览器可调用的在线推理服务；另一方面，它将原始影像、检测框、病害类别、置信度、面积、网格位置、报告内容和操作日志统一纳入数据流，使历史建筑清水砖墙的每一次巡检都能沉淀为可管理的数字资产。

## 2.4.2 平台总体架构与技术组成

系统采用典型 B/S 架构，整体可划分为前端交互层、后端业务服务层、模型推理层和数据资产层四个部分。前端负责用户操作、图片上传、参数配置、检测结果展示、热力图渲染和报告导出；后端负责文件接收、权限管理、模型调用、结果解析、面积计算、报告组织和记录持久化；模型推理层负责执行 YOLOv11 病害识别，并输出病害类别、置信度和空间坐标；数据资产层负责保存检测历史、系统配置、日志记录和立面任务 JSON 数据。

前端工程采用 Vue 3 组合式 API 与 TypeScript 开发，通过 Element Plus 组件库实现上传控件、表单配置、弹窗提示、统计卡片、标签、按钮、折叠面板和报告展示等界面。系统主页面集成了拍摄指南、图片上传、模型参数设置、检测结果展示、统计看板、修缮报告和立面普查等功能区。前端 `api` 服务层统一封装了 `/api/detect`、`/api/report/generate`、`/api/facade/upload`、`/api/facade/analyze/:jobId`、`/api/facade/report/:jobId`、`/api/quality-check` 等接口，使页面逻辑和后端服务保持清晰解耦。

后端工程基于 Express 构建。Multer 用于处理单图和全景大图上传，Sharp 用于读取图像尺寸、生成立面切片、进行图像预处理，Node.js 子进程用于调用 Python 推理脚本，认证模块用于用户登录、注册、历史记录、系统日志和系统设置管理。服务启动后统一对外提供 API 端点，并可在生产环境下直接托管前端构建产物，实现单服务部署。

模型推理层采用 YOLOv11 红砖墙病害检测模型作为核心视觉引擎。系统支持 Python 脚本推理模式，并预留 ONNX Runtime Node.js 推理模块。ONNX 模块可加载 `best.onnx` 权重，完成 640×640 letterbox 预处理、张量构建、模型前向推理、类别分数解析、坐标还原和 NMS 后处理。该设计使平台后续可根据部署环境在 Python 推理、本地 ONNX 推理和云端 PAI-EAS 推理之间灵活切换。

## 2.4.3 用户登录、系统导航与权限管理模块

为了满足工程项目中多用户协同使用和数据追溯需求，平台开发了基础认证与系统管理模块。系统提供登录、注册、退出登录和用户信息查询接口，登录成功后由后端生成 token，并在用户后续访问历史记录、系统日志、系统设置等管理接口时进行鉴权。平台默认初始化管理员账号，同时允许普通用户注册使用，从而适应课题演示、现场测试和后续多角色管理场景。

登录模块记录用户登录成功、登录失败、注册成功和退出登录等行为，并写入系统日志。日志中包含用户名、操作类型、操作内容、IP 地址、请求地址、请求方法、操作状态和时间信息。该设计不仅满足基本身份识别，也为后续将平台接入企业 OA 或 SSO 单点登录系统预留了数据基础。

在前端交互上，系统采用统一布局页面组织业务入口，包括首页检测、历史记录、系统日志和系统设置等页面。查勘人员主要使用首页检测和报告功能，管理人员可通过历史记录与系统日志进行追溯，管理员可在系统设置中维护模型版本、置信度阈值、热力图开关、自动报告开关、存储类型、缓存策略等参数。通过这种模块化导航，平台从单一检测页面扩展为具备基础管理能力的工程应用系统。

## 2.4.4 首页单图查勘功能与标准化采集流程

首页是平台最核心的业务入口，主要服务于红砖墙病害的单图快速检测。用户在现场按照“标准拍摄三步法”获取墙面照片后，可直接在首页上传 JPG 或 PNG 格式图片。平台对单图上传设置了格式和大小约束，默认支持 JPG、JPEG、PNG 格式，单张图片大小控制在 10MB 以内，从源头避免非图像文件和过大文件造成服务异常。

上传图片后，系统立即生成本地预览，并自动执行图像质量初筛。该质量检测基于浏览器 Canvas 完成，不依赖后端推理资源。系统首先将图片缩放到较小尺寸进行快速采样，然后计算灰度平均值评估亮度水平，利用拉普拉斯算子方差估计图像模糊程度，并结合图像分辨率和灰度对比度判断是否存在拍摄过远、细节不足等问题。当图像存在过暗、过曝、模糊、分辨率过低或距离过远风险时，平台会弹出质量提示，并给出重新拍摄建议。

该质量初筛机制与课题提出的“标准拍摄三步法”相互配合。对于现场人员而言，系统提示不只是技术警告，而是操作规范的即时反馈。例如，当系统提示“距离可能太远，砖块细节不清晰”时，用户可根据建议靠近墙面约 2 米，使画面横向容纳约 10 块红砖；当系统提示“光线过强/过曝”时，用户可避开强光直射或选择阴天、早晚时段重新拍摄。通过这种前端交互设计，平台把数据采集质量控制嵌入实际查勘流程，减少因低质量输入导致的漏检和误检。

在检测参数方面，首页提供参考砖长度和模型推理参数设置。参考砖长度默认值为 240mm，用于后续基于像素比例估算病害面积；模型参数包括置信度阈值、IoU 阈值和推理图片尺寸。默认置信度阈值为 0.30，IoU 阈值为 0.45，推理尺寸为 640。对于细小裂缝或复杂纹理场景，可适当调整参数以平衡召回率和误检率。该设计使平台既能面向普通用户快速使用，也能为研究人员提供一定的模型调试空间。

## 2.4.5 YOLOv11 推理接口与五类病害识别结果解析

单图检测由前端调用 `/api/detect` 接口完成。前端将图片文件、参考砖长度、模型置信度阈值、IoU 阈值和推理尺寸封装为 `FormData` 提交至后端。后端接收请求后，保存上传图片，读取图像宽高，并根据运行环境选择本地 YOLOv11 推理、ONNX 推理或演示模式。模型推理完成后，后端将原始模型输出转换为平台统一的检测结果格式。

平台在类别语义上进行了严格统一。模型训练类别包括 `01:LF`、`02:QS`、`03:P`、`04:B-FH`、`05:B-FJ`，系统分别映射为裂缝、缺损、植物附着、风化、泛碱五类中文病害名称。这一映射同时应用于 Python 推理脚本、ONNX 推理模块、后端统计逻辑和前端显示组件，避免模型标签、报告文本和页面展示之间出现不一致。

检测结果的核心字段包括病害编号、类别名称、原始类别名、置信度、边界框坐标、面积、严重程度、网格编号、原图尺寸、检测时间、模型信息、标注图路径和坐标文本内容等。其中边界框坐标用于前端叠加显示，置信度用于判断模型识别可靠性，面积与严重程度用于后续报告生成，模型信息用于记录检测来源和版本。系统还会按照病害类型生成汇总结果，包括每类病害的检出数量、总面积、最高严重程度和实例列表。

前端在展示检测结果时，根据图片真实宽度和页面显示宽度计算缩放比例，将后端返回的坐标动态映射到页面图片上。平台采用固定颜色体系区分五类病害：风化为红色，泛碱为蓝色，裂缝为橙色，植物附着为紫色，缺损为青绿色。用户可以在图像上直接看到每一处病害的位置、类别和置信度，也可以通过统计卡片查看总体数量和面积指标。该展示方式使 AI 输出结果具备可解释性和工程可读性。

## 2.4.6 检测结果看板与前端可视化展示

为了使检测结果不只是停留在标注图片层面，平台在前端构建了统计看板和结果摘要。单图检测完成后，系统根据 `summary` 汇总数据展示各类病害数量、面积和严重程度，用户可以快速判断当前墙面照片中主要病害类型及其风险水平。对于风化和泛碱等适合面积统计的病害，平台重点展示其累计受损面积；对于裂缝、植物附着和缺损等病害，则结合数量、位置和严重程度进行展示。

平台同时支持标注图和原图叠加两类查看方式。标注图用于快速查看模型生成的整体识别效果，原图叠加模式则允许系统基于坐标框动态绘制病害边界，方便用户在不同显示尺寸下保持定位准确。检测结果还可输出坐标文本内容，便于后续复核模型输出或与其他图像处理软件进行数据对接。

为了适配移动端和现场使用场景，前端页面对响应式布局进行了优化。系统监听窗口尺寸变化，并在图片加载或页面缩放后重新计算检测框比例，保证在桌面浏览器、平板或手机浏览器中均能正确显示病害位置。对于现场查勘人员而言，这意味着平台可以直接作为轻量化移动 Web 工具使用，不依赖专用客户端。

## 2.4.7 整墙立面普查模式与大图切片分析

除单张照片检测外，平台开发了“立面普查模式”，用于对历史建筑整墙正射影像或全景立面图进行整体病害评估。该模式特别适用于静安别墅等具有连续红砖立面的历史建筑场景，可将单点查勘扩展为整墙空间化诊断。

在前端操作流程上，用户首先上传全景立面影像，并填写项目名称、墙面名称、墙体实际宽度、墙体实际高度和网格尺寸。系统默认项目名称可设置为“静安别墅红砖病害智能检测系统”，墙面名称可设置为具体待查勘立面，网格尺寸默认按 1m 进行划分。平台对全景文件设置更高容量上限，支持 JPG、PNG、TIFF、WEBP 等格式，满足正射大图和高分辨率影像输入需求。

后端接收全景图后，会创建一个独立的立面任务，并生成唯一 `jobId`。任务记录中保存项目名称、墙面名称、原图路径、原图 URL、图像像素宽高、墙体实际尺寸、像素到米的换算比例、网格大小、切片尺寸、重叠率、任务状态、处理进度、切片列表、检测结果、网格结果和统计汇总等数据。该任务化设计使整墙分析具备可恢复、可查询和可归档能力。

由于整墙正射图往往尺寸较大，无法直接一次性输入 YOLOv11 模型，平台采用切片推理策略。系统使用 Sharp 将大图按默认 1280 像素切片，并设置约 15% 的重叠率，以减少病害位于切片边界时被截断的问题。每个切片记录包含切片编号、行列索引、全局偏移量、宽高、文件路径、访问 URL 和处理状态。后端逐一调用模型推理服务检测每个切片，再将切片内的局部坐标恢复为整墙全局坐标。

该流程实现了“大图局部精细识别”和“整墙全局统一定位”的结合。对于模型而言，每个切片保持适合识别的分辨率；对于工程应用而言，所有检测结果最终仍回到整墙统一坐标系中，可用于计算真实面积、生成热力图和定位重点修缮区域。

## 2.4.8 网格化空间统计与病害热力图展示

在整墙立面普查中，平台将墙体实际尺寸与图像像素尺寸建立对应关系，计算 `pixelToMeterX` 和 `pixelToMeterY` 两个尺度参数。系统根据用户输入的网格尺寸自动划分墙面网格，例如按 1m×1m 生成 `R1-C1`、`R1-C2` 等网格编号。每个病害实例根据其全局边界框中心点归属到对应网格，进而形成空间化统计结果。

每个网格记录包含网格编号、行列索引、实际空间坐标、网格宽高、病害总数、受损面积、裂缝长度、病害类型分布、关联检测实例、关联切片编号和风险强度值。系统根据网格内病害面积和病害数量综合计算 `intensity`，用于表示该区域的病害密集程度。强度越高，说明该网格越应作为重点修缮或复核对象。

前端通过 `FacadeHeatmapCanvas` 组件在整墙原图上绘制热力图。系统先绘制原始立面影像，再根据每个网格的 `intensity` 叠加半透明红色热力层，并绘制白色网格线。用户移动鼠标时可查看悬浮提示，包括网格编号、病害数量和受损面积；点击某个网格后，右侧整墙统计看板会显示该网格的病害数量、受损面积和裂缝长度，并支持打开原始切片查看局部细节。

整墙统计看板集中展示病害总数、总受损面积、裂缝长度和高风险网格数量等关键指标。对于管理人员而言，热力图可以直观回答“病害主要集中在哪里”“哪几个区域需要优先修缮”“整墙受损面积大约是多少”等问题。相比传统逐张图片翻看，网格化热力图显著提高了整墙病害普查的空间认知效率。

## 2.4.9 修缮报告自动生成与多格式导出

平台将检测结果进一步转化为修缮报告，使 AI 识别结果能够直接服务工程决策。单图检测完成后，用户点击“生成修缮报告”，前端将完整检测结果提交至 `/api/report/generate` 接口。后端根据病害汇总、检测数量、面积、严重程度和网格信息生成结构化报告数据，再由前端报告组件渲染为可阅读的诊断报告。

单图修缮报告包括总体评估、五类病害详情和修缮优先级计划三个主要部分。总体评估部分列出病害种类、检出总数、风化面积、泛碱面积、综合风险等级和总体建议。系统根据检出数量和病害类型数量自动划分低风险、中风险和高风险：病害数量较少时建议定期巡检，病害数量较多或类型复杂时建议尽快制定修缮计划，达到高风险条件时建议立即开展修缮工作。

五类病害详情部分覆盖风化、泛碱、裂缝、植物附着和缺损。系统为每类病害预置了病害描述、严重程度说明、修缮方法、修缮材料和预防措施，并结合实际检测结果动态填入检出数量、受损面积、最高严重程度和工程量单位。例如，风化和泛碱以平方米作为主要计量单位，裂缝、植物附着和缺损则可按处数进行统计。报告还根据严重程度将病害划分为紧急处理、重点关注和常规维护三个优先级层次。

前端报告组件支持 TXT、Word 和 PDF 等导出方式。Word 导出适用于项目汇报和结题材料整理；PDF 导出适用于正式归档；TXT 导出便于二次编辑与系统集成。通过多格式导出，平台将 AI 检测结果转化为能够进入工程文件流转体系的正式资料。

对于整墙立面普查，平台提供 `/api/facade/report/:jobId` 接口生成整墙修缮报告。该报告基于立面任务数据，输出墙体面积、总受损面积、裂缝长度、高风险网格数量、病害面积占比、综合风险、重点修缮网格和各类病害数量等确定项。整墙报告能够直接服务修缮范围划定、分区施工计划制定和网格化复核。

## 2.4.10 历史记录、系统日志与数据可追溯机制

为了保证每一次查勘结果都能被追溯和复核，平台开发了历史记录和系统日志模块。系统初始化时会在后端数据目录中创建用户数据、历史记录、系统日志和系统设置等 JSON 文件。每次检测完成后，平台可将检测编号、检测时间、项目名称、图片名称、病害类型、病害数量、检测状态、结果路径和报告路径等信息写入历史记录，形成按时间排序的查勘档案。

历史记录接口支持按项目名称、病害类型、检测状态和时间范围进行筛选，并支持分页查询、查看详情、删除记录和报告生成等操作。该功能适用于工程项目中对多次检测结果进行集中管理，例如对同一历史建筑在不同日期、不同墙面、不同楼层的检测结果进行分类查询和复核。

系统日志模块记录平台运行中的关键操作，包括登录日志、检测日志、系统日志和导出日志。日志字段包括日志编号、操作时间、用户名、日志类型、操作内容、IP 地址、状态、请求地址、请求方法和消息说明。后端保留最近 1000 条日志，既避免日志无限增长，又能满足平台日常审计需求。管理人员可根据用户名、日志类型、操作状态和时间范围进行检索，快速定位异常操作或重要业务行为。

系统设置模块则提供平台参数管理能力。可配置内容包括系统名称、系统版本、模型版本、默认置信度阈值、是否启用热力图、是否启用自动报告、存储类型、缓存上限和缓存策略等。平台还提供系统状态接口，用于查看服务器运行状态、模型连接状态、数据存储状态和最后更新时间。该模块使平台具备一定运维管理能力，而不是单纯的算法调用页面。

## 2.4.11 数据资产集成与标准化数据结构

平台的数据资产集成体现在对“原始数据、过程数据、结果数据和管理数据”的统一组织。原始数据包括现场拍摄的单图、整墙正射影像和切片图像；过程数据包括上传时间、模型参数、图像质量检测结果、切片编号、坐标偏移、任务状态和推理进度；结果数据包括病害类别、置信度、边界框、多边形、面积、裂缝长度、网格强度和修缮建议；管理数据包括用户、历史记录、系统日志、系统设置和导出报告。

单图检测数据以 `DetectionResult` 为核心结构，包含图像路径、图像宽高、检测总数、检测实例列表、病害汇总、检测时间、模型信息、参考砖长度、标注图地址和推理参数。每个检测实例包含类别、置信度、坐标、面积、严重程度和网格编号。该结构既适合前端展示，也适合报告生成和后续归档。

立面普查数据以 `FacadeJob` 为核心结构，包含任务编号、项目名称、墙面名称、源图路径、图像尺寸、实际墙体尺寸、像素—米换算比例、网格大小、切片大小、重叠率、任务状态、切片数组、检测数组、网格数组、统计汇总和更新时间。该结构完整描述了一次整墙分析的输入、处理过程和输出结果，可作为后续复核、重新分析和多期对比的基础。

报告数据则将检测结果进一步转化为工程语义结构，包括报告标题、生成时间、检测时间、总体评估、病害详情、修缮计划、重点网格和损伤占比等确定字段。通过这种多层数据组织方式，平台实现了从“照片”到“检测结果”、从“检测结果”到“工程报告”、从“工程报告”到“历史档案”的连续转化。

当前平台采用本地 JSON 文件进行轻量化持久化，便于课题开发、部署和演示。由于各类数据结构已经标准化，后续可平滑迁移至 MySQL、PostgreSQL、MongoDB 或对象存储平台，并可进一步与阿里云 OSS、PAI-iTAG 标注数据和 PAI-EAS 推理服务形成完整云端数据闭环。

## 2.4.12 AI 智能分析与扩展能力预留

除传统规则化报告外，平台还预留了 AI 智能分析服务。后端实现了面向检测结果的 AI 分析接口，可根据病害统计、面积、严重程度和风险等级构造提示词，优先调用 PAI LLM 服务；若未配置 PAI LLM，则可回退到 SiliconFlow 大模型服务；当外部大模型不可用时，系统仍可使用规则引擎和预设建议提供基础分析结果。

该功能为平台从“检测工具”升级为“智能诊断助手”提供了接口基础。未来可围绕病害成因分析、修缮工艺推荐、材料用量估算、巡检周期建议和风险趋势解释等方向进行扩展。例如，当系统检测到泛碱面积较大且集中在墙脚区域时，AI 分析模块可结合潮湿环境、盐分迁移和排水条件给出更具解释性的诊断建议；当裂缝集中出现在门窗洞口附近时，可提示关注局部应力集中或结构沉降问题。

平台还预留了序列影像自动合成接口。当前整墙立面分析主要采用 Reality Capture 离线生成正射影像后上传，后续可接入云端影像拼接、正射校正或三维重建服务，使用户能够直接上传多张现场照片，由系统自动生成立面拼接图并进入病害识别流程。该扩展方向可进一步降低整墙普查的数据生产门槛。

## 2.4.13 平台部署运行与工程化特点

平台工程以 `brick-wall-detector` 为项目主体，前后端代码统一管理。项目依赖包括 Vue、Vite、Element Plus、ECharts、Axios、Express、Multer、Sharp、dotenv 等，运行环境要求 Node.js 18 以上。开发环境下可通过并行启动前端 Vite 服务和后端 Express 服务进行调试；生产环境下可先构建前端静态资源，再由后端服务统一托管。

服务启动后，后端会输出系统运行信息，包括服务地址、调试模式、仓库根目录、服务目录、目标检测模型状态、AI 大模型服务状态和 API 端点列表。平台核心 API 包括单图病害检测、报告生成、AI 智能分析、AI 对话、AI 服务状态、拍摄质量初筛、立面全景上传、立面普查分析、立面任务查询、整墙修缮报告和健康检查等。这种启动信息有助于部署人员快速判断系统是否正常运行。

在工程化设计上，平台具备三方面特点。第一，业务链路完整，从图片上传到报告导出均已形成闭环。第二，部署方式轻量，不依赖复杂客户端，浏览器即可访问。第三，扩展性较强，模型服务、AI 分析服务、影像拼接服务和数据存储方式均可替换或升级。对于课题结题而言，该平台不仅证明了 YOLOv11 模型的识别能力，也证明了其能够被封装为实际可用的数字化查勘系统。

## 2.4.14 平台开发成效与应用价值

通过平台开发，课题组完成了红砖墙病害检测模型从算法实验到工程系统的转化。系统能够识别裂缝、缺损、植物附着、风化和泛碱五类典型病害，并通过统一颜色、统一类别、统一字段和统一报告模板进行展示和输出。现场人员可通过单图模式完成快速查勘，管理人员可通过立面普查模式掌握整墙病害空间分布，项目负责人可通过修缮报告获取风险等级和处理建议。

从工程效率看，平台显著减少了人工整理工作。传统查勘需要人工拍照、人工标注、人工统计、人工撰写报告，而平台能够自动完成模型识别、数量统计、面积估算、风险判断和报告生成，降低了查勘工作的人力成本和时间成本。从技术应用看，平台将复杂的深度学习模型封装为直观的 Web 工具，使非算法背景的工程人员也能直接使用 AI 检测能力。

从历史建筑保护看，平台为红砖墙病害的预防性保护提供了数字化底座。每次巡检结果都可以形成带时间、位置、图像、病害类型、面积和报告的数据记录；多次巡检结果可以用于比较病害扩展趋势；低置信度、复杂光影和难识别样本可以回流至标注与训练流程，推动模型持续迭代。由此，平台不仅服务当前项目查勘，也为后续建立历史建筑健康档案和病害演化数据库奠定了基础。

综上，“红砖墙病害智能检测 Web 系统”实现了标准化采集、智能化识别、可视化展示、定量化评估、报告化输出和资产化管理的全过程集成。该平台是本课题“基于机器视觉的 AI 智能损伤诊断评估模型”从研究成果走向工程应用的关键载体，为历史建筑清水砖墙的日常巡检、修缮决策和长期保护提供了低门槛、高可用、可追溯的数智化工具。
